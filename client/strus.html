Description of this demo search

  • This is a demo system that has been built to show the performance of strus under conditions of limited resources. Older hardware has been used, because a system pushed to its limits is better to judge. Because the queries will not be very frequent on this system, the chance that the engine has to access disk to retrieve the result, is very high. The size of the index is 6 times bigger than the memory available. Be aware of the size of the index (15'347'649 documents). Normally a search index of this size would be split into smaller units. If somebody thinks strus is slow, so be it. This demo tries to address a public that is aware of what is happening here.

    • Hardware description

    1. Motherboard: Intel S3000AHV
    2. Processor: 1 Intel Xeon Processor 3040, 1.86 GHz
    3. Memory: 8 GB DDR2 RAM, 533 MHz
    4. Hard disc: Western Digial 2TB SATA, product number: WDC WD2002FYPS-0
    5. Index size on disc: ~50G

    Weighting schemas:

    1. BM25: classical BM25 with k1=0.75, b=2.1, avgdoclen=500
    2. BM25_dpfc: BM25 (k1=0.75, b=2.1, avgdoclen=500) with accumulated increments added to the ff for discrete proximity conditions fulfilled.
      1. Increment of 4.0 for appearance in title
      2. Increment of 3.0 if followed by the subsequent term in the query
      3. Increment of 1.5 if followed by the previous term in the query
      4. Increment of 0.5 if another term in the query appears in the same sentence
      5. Increment of 0.25 if another term in the query appears in the following sentence
      6. Increment of 0.125 if another term in the query appears in the sentence after next

    In both schemas there is an extra weight added for popularity: w=tanh(n/100), where n is the number of links pointing to the page (df of references)

    In strus what you weight and how you weight is distinguished. The what you weight is an expression that matches if all query terms match somewhere in the document. Documents were one feature of the query does not match, are not appearing in the result.

    Problems/Bugs:

    1. Due to a bug in a filter that was implemented to remove features not supposed to be searched like for example "abababab12121212ababa", years with 4 digits were accidentally removed from the index. You cannot search for years like "1970". The bug will be fixed with the next build end of June.
    2. The BM25_dpfc weighting scheme is currently very vulnerable to searches including very frequent words. For queries containing such words, answering times will not be acceptable. You have to choose BM25 to jugde the preformance of the search engine. I'll try to fix this as soon as possible.
